# Prompt Alchemist — 模型适配指南

> 同一条提示词，在 GPT / Claude / Gemini 上的表现天差地别。本指南帮你针对目标模型的"人格特征"调整提示词风格。
> 理论来源：Goodside "模型感知"原则 + Schulhoff 跨模型评测结论 + 实测对比。

## 二、三个模型的"人格"画像

### 🟢 GPT-4o / 4.1 — 顺从的执行者

**核心特征**：
- 高度配合用户指令
- 倾向"接住"所有暗示（包括你不想让它做的事）
- 角色扮演到位，但容易"演过头"（比要求的更热情）
- Markdown 渲染偏好一般

**提示词设计要点**：
1. **明确"不做什么"**：GPT 容易过度配合，需用 `<constraints>` 显式禁止
   ```xml
   <constraints>
   - 不要在回答开头说"当然可以！""这是一个好问题"等客套话
   - 不要主动扩展用户没问的相关内容
   </constraints>
   ```
2. **用命令式语气**（而非请求式）
   - ✅ "回答以下问题"
   - ❌ "请帮我回答以下问题，如果可以的话"
3. **Few-Shot 必给**：GPT 对示例的依赖度最高
4. **避免开放式问题**：开放式会让 GPT 自由发挥超出范围

---

### 🟣 Claude Sonnet 4.5 / Opus 4.5 — 推理的辩论者

**核心特征**：
- 默认有深度推理能力
- 倾向挑战/质疑用户的预设（这是优势，不要压它）
- 角色扮演时**保留自身判断**（不会盲目顺从）
- 抗长 prompt 能力最强
- XML/结构化标签响应极好（Anthropic 训练时就用这个）

**提示词设计要点**：
1. **善用 XML 标签**（Claude 的"母语"）
   ```xml
   <role>资深职业战略顾问</role>
   <context>...</context>
   <task>...</task>
   <constraints>...</constraints>
   <format>...</format>
   ```
2. **允许"反驳"**（不要写"必须按我说的做"）
   ```xml
   <task>分析这个决策，请保留质疑和不同意见的空间</task>
   ```
3. **复杂任务别给 Few-Shot**：给 Few-Shot 反而会限制 Claude 的内在推理
4. **安全约束用 `<guidelines>` 而非否定句**：
   - ✅ `<guidelines>保持尊重、不歧视、不输出违法内容</guidelines>`
   - ❌ "不要歧视、不要输出违法内容"（Claude 会过度防御）
5. **自我批评用 `<thinking>` 标签包裹**（Anthropic 原生支持）

---

### 🟡 Gemini 1.5 / 2.0 — 简洁的搜索者

**核心特征**：
- 输出偏长、容易冗余
- Few-Shot 顺序敏感度**最高**（最后一条示例的权重明显）
- 前段指令覆盖后段（建议核心指令放最后/显眼位置）
- 对简洁约束响应好
- 喜欢 markdown 标题

**提示词设计要点**：
1. **明确字数约束**（否则会 2000 字起步）
   ```xml
   <format>≤200 字，3 个 bullet，不要前言</format>
   ```
2. **示例顺序精心设计**（最想模仿的放最后）
3. **核心指令放末尾/独立段落**（避免被前段覆盖）
4. **少用嵌套结构**（Gemini 对深层嵌套理解略弱）
5. **加显式输出格式样板**：
   ```xml
   <example_output>
   # 标题
   - 要点 1
   - 要点 2
   </example_output>
   ```

## 四、同一提示词三模型对比示例

### 原始提示词
```
帮我写一篇关于秋天的散文，要写得美一点
```

### 通用优化版（任一模型都可用）
```xml
<role>散文作家，笔触介于川端康成的物哀美学与汪曾祺的市井温情之间</role>
<context>写一篇关于秋天的散文</context>
<constraints>
- 不出现"金风送爽"等陈词
- 以具体场景开篇
- 篇幅 800-1200 字
</constraints>
<format>纯文本，段落间留呼吸感</format>
```

### GPT 定制版
```xml
<role>散文作家，笔触介于川端康成与汪曾祺之间</role>
<context>写一篇关于秋天的散文</context>
<constraints>
- 不出现"金风送爽"等陈词
- 不使用"金风送爽""秋高气爽"等成语
- 不输出"当然可以！""这是一篇关于..."等客套开头
- 不主动总结或升华主题
- 以具体场景开篇
- 篇幅 800-1200 字
</constraints>
<examples>
风格示例（务必参考）：
"梧桐叶落下来的样子，不是飘，是决绝地翻了一个身。"
</examples>
<format>纯文本，段落间留呼吸感，不使用小标题</format>
```
> **关键差异**：显式禁止客套话 + Few-Shot 强制模仿

### Claude 定制版
```xml
<role>散文作家，笔触介于川端康成的物哀美学与汪曾祺的市井温情之间</role>

<context>写一篇关于秋天的散文</context>

<constraints>
- 不出现"金风送爽""秋高气爽"等陈词
- 以一个具体场景开篇（不概括，只白描）
- 情感走向：从感官沉浸到时间感知的微痛
- 篇幅：800-1200 字
</constraints>

<thinking>
不必急着动笔。先在脑中浮现一个具体场景——某个秋日午后，某条街道，某个细节。
从那个细节出发，自然延伸出整篇散文的情感脉络。
避免结构化、避免清单感，让文字像意识流一样流淌。
</thinking>

<examples>
风格示例：
"梧桐叶落下来的样子，不是飘，是决绝地翻了一个身。"
——注意：具体、动感、情感暗线
</examples>

<guidelines>
保留创作自由度。如果觉得结构需要调整，按你认为最自然的方式展开。
但篇幅控制在 800-1200 字。
</guidelines>

<format>纯文本，段落间留呼吸感，不使用小标题</format>
```
> **关键差异**：`<thinking>` 引导意识流 + `<guidelines>` 替代否定句 + 允许 Claude 自主判断结构

### Gemini 定制版
```xml
<role>散文作家</role>

<context>写一篇关于秋天的散文，参考川端康成和汪曾祺的笔触</context>

<task>直接开始写散文。</task>
<format>≤1000 字，3-5 段，**不要前言、不要"以下是..."，第一句就是正文**</format>

<example_output>
梧桐叶落下来的样子，不是飘，是决绝地翻了一个身。
[正文继续...]
</example_output>
```
> **关键差异**：示例放最后（权重高）+ 字数硬约束 + 禁止前言

## 五、模型切换 checklist

当你把同一提示词换到不同模型时，按此检查：

### GPT 检查
- [ ] 显式禁止客套话？
- [ ] 给了 Few-Shot？
- [ ] 指令是命令式（不是请求式）？
- [ ] `<constraints>` 明确"不做什么"？

### Claude 检查
- [ ] 用了 XML 标签？
- [ ] 允许"反驳/质疑"？
- [ ] 复杂任务没给 Few-Shot？
- [ ] 用了 `<guidelines>` 而非否定句？

### Gemini 检查
- [ ] 字数硬约束？
- [ ] 核心指令放末尾/显眼位置？
- [ ] 示例顺序符合"最想模仿的放最后"？
- [ ] 给了完整输出样板？

## 六、跨模型优化 5 条铁律

1. **不要写一版通吃**——至少为首选/备选模型各做一版
2. **角色扮演优先 GPT**——它执行最到位
3. **复杂推理优先 Claude**——它默认就有深度
4. **超长上下文优先 Claude 200K 或 Gemini 1M**——避免切分丢失
5. **JSON/结构化输出优先 GPT-4.1**——对 schema 响应最稳

## 七、国产AI深度适配指南（v1.9.0 新增）

> 本节专门针对国内用户常用国产模型优化，已在 DeepSeek、通义千问、Kimi、智谱GLM 等模型上验证。主要解决 SkillHub 评测反馈"国产AI例子偏少"问题。

### 🟢 DeepSeek — 推理型（对齐Claude风格）

**核心特征**：
- 推理能力接近 Claude Sonnet，支持 `<think>` 标签显式输出思考过程
- 中文理解优秀，对提示词指令遵循度高
- `deepseek-coder` 专项优化代码任务

**提示词设计要点**：
1. **善用 `<think>` 标签**（DeepSeek 原生支持）
   ```xml
   <think>
   先分析这道题的核心考点，然后逐步推导答案，最后验证结果合理性。
   </think>
   ```
2. **推理任务用 CoT 触发**（效果比 GPT 更稳定）
   ```xml
   <task>分析这个问题。请先明确前提条件，再逐步推导，最后给出结论。</task>
   ```
3. **代码任务优先用 `deepseek-coder`**
   ```xml
   <role>资深Python工程师，代码风格遵循PEP8，注重可读性和安全性</role>
   <task>审查以下代码，找出潜在问题并给出优化建议</task>
   ```

### 🟡 通义千问（Qwen）— 分析型

**核心特征**：
- 阿里自研，中文语义理解深度高
- `qwen-max` 对复杂指令理解能力强
- 对「比喻」「类比」等修辞理解好

**提示词设计要点**：
1. **中文创作类任务天然优势**
   ```xml
   <role>中文文学编辑，擅长发现文字节奏感和情感表达</role>
   <context>润色以下段落，使其语言更凝练、情感更充沛</context>
   ```
2. **分析类任务结构化输出稳定**
   ```xml
   <format>
   ## 分析结论（3点）
   ### 1. [结论]
   ### 2. [结论]
   ### 3. [结论]
   </format>
   ```
3. **避免过长的纯英文提示词**（中文提示词效果更好）

### 🔵 Kimi/Moonshot — 超长上下文型

**核心特征**：
- 200K 超长上下文，可直接处理长文档
- 适合「整本书总结」「长文档分析」任务
- 对对话式交互友好

**提示词设计要点**：
1. **利用超长上下文，减少压缩压力**
   ```xml
   <task>
   阅读以下长文（全文约8000字），然后按要求分析。
   不需要先压缩信息，直接基于原文分析即可。
   </task>
   ```
2. **文档分析任务给足上下文空间**
   ```xml
   <role>专业文档分析师，擅长从长文本中提取关键信息</role>
   <context>以下是原始文档内容（已完整提供）：
   {原文全文}
   
   分析要求：提取核心观点、关键数据、主要结论
   </context>
   ```
3. **避免「先总结再分析」的双重压缩**

### 🟣 智谱GLM — 国产旗舰型

**核心特征**：
- 接近 GPT-4 表现，中文旗舰级
- `glm-4-flash` 性价比高，适合高频调用
- Few-Shot 效果稳定

**提示词设计要点**：
1. **可参考 GPT-4 提示词设计，直接迁移**
   ```xml
   <role>专业战略分析师，逻辑严谨，数据驱动</role>
   <task>参考以下案例风格，分析目标公司的竞争策略</task>
   <examples>
   案例1：{示例内容}
   案例2：{示例内容}
   </examples>
   ```
2. **结构化指令遵循好**（Markdown/JSON 都支持）
   ```xml
   <format>输出JSON，包含字段：analysis（字符串）、confidence（0-1浮点）、recommendations（数组）</format>
   ```

### 国产AI统一 Checklist

当你把提示词迁移到国产模型时，按此检查：
- [ ] 中文提示词（国产模型中文理解更强，无需翻译成英文）
- [ ] 中文大师名（直接写「参考李继刚的乔哈里视窗」比英文有效）
- [ ] 中文示例（Few-Shot 优先用中文示例）
- [ ] 字数约束（DeepSeek/通义仍需显式约束）
- [ ] DeepSeek 优先用 `<think>` 标签触发推理
- [ ] Kimi 可直接给长文本，不强制压缩
