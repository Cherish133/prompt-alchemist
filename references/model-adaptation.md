# Prompt Alchemist — 模型适配指南

> 同一条提示词，在 GPT / Claude / Gemini 上的表现天差地别。本指南帮你针对目标模型的"人格特征"调整提示词风格。
> 理论来源：Goodside "模型感知"原则 + Schulhoff 跨模型评测结论 + 实测对比。

---

## 一、核心结论速查表

| 维度 | GPT-4o/4.1 | Claude Sonnet/Opus | Gemini 1.5/2.0 |
|------|------------|-------------------|---------------|
| **人格倾向** | 顺从型——会"接住"用户的指令 | 推理型——会挑战/质疑/补充 | 综合型——倾向长输出 |
| **指令风格** | 命令式（"做 X"） | 推理式（"考虑 X 是否真的合理"） | 简洁约束式（"≤200字"） |
| **角色扮演** | 强响应（执行到位） | 中响应（保留自身判断） | 弱响应（容易"出戏"） |
| **Few-Shot 敏感度** | 高 | 中（更看内在推理） | 高 |
| **示例顺序影响** | 中 | 高 | 极高（容易过度模仿最后一条） |
| **格式遵从度** | 中（容易"自由发挥"） | 高（结构化标签响应好） | 中（喜欢 markdown） |
| **CoT 思维链** | 必须显式说"step by step" | 默认就有深度推理 | 显式触发更稳定 |
| **过长提示词** | 后段容易丢失 | 抗压强（200K 上下文） | 容易"前段覆盖" |

## 二、三个模型的"人格"画像

### 🟢 GPT-4o / 4.1 — 顺从的执行者

**核心特征**：
- 高度配合用户指令
- 倾向"接住"所有暗示（包括你不想让它做的事）
- 角色扮演到位，但容易"演过头"（比要求的更热情）
- Markdown 渲染偏好一般

**提示词设计要点**：
1. **明确"不做什么"**：GPT 容易过度配合，需用 `<constraints>` 显式禁止
   ```xml
   <constraints>
   - 不要在回答开头说"当然可以！""这是一个好问题"等客套话
   - 不要主动扩展用户没问的相关内容
   </constraints>
   ```
2. **用命令式语气**（而非请求式）
   - ✅ "回答以下问题"
   - ❌ "请帮我回答以下问题，如果可以的话"
3. **Few-Shot 必给**：GPT 对示例的依赖度最高
4. **避免开放式问题**：开放式会让 GPT 自由发挥超出范围

---

### 🟣 Claude Sonnet 4.5 / Opus 4.5 — 推理的辩论者

**核心特征**：
- 默认有深度推理能力
- 倾向挑战/质疑用户的预设（这是优势，不要压它）
- 角色扮演时**保留自身判断**（不会盲目顺从）
- 抗长 prompt 能力最强
- XML/结构化标签响应极好（Anthropic 训练时就用这个）

**提示词设计要点**：
1. **善用 XML 标签**（Claude 的"母语"）
   ```xml
   <role>资深职业战略顾问</role>
   <context>...</context>
   <task>...</task>
   <constraints>...</constraints>
   <format>...</format>
   ```
2. **允许"反驳"**（不要写"必须按我说的做"）
   ```xml
   <task>分析这个决策，请保留质疑和不同意见的空间</task>
   ```
3. **复杂任务别给 Few-Shot**：给 Few-Shot 反而会限制 Claude 的内在推理
4. **安全约束用 `<guidelines>` 而非否定句**：
   - ✅ `<guidelines>保持尊重、不歧视、不输出违法内容</guidelines>`
   - ❌ "不要歧视、不要输出违法内容"（Claude 会过度防御）
5. **自我批评用 `<thinking>` 标签包裹**（Anthropic 原生支持）

---

### 🟡 Gemini 1.5 / 2.0 — 简洁的搜索者

**核心特征**：
- 输出偏长、容易冗余
- Few-Shot 顺序敏感度**最高**（最后一条示例的权重明显）
- 前段指令覆盖后段（建议核心指令放最后/显眼位置）
- 对简洁约束响应好
- 喜欢 markdown 标题

**提示词设计要点**：
1. **明确字数约束**（否则会 2000 字起步）
   ```xml
   <format>≤200 字，3 个 bullet，不要前言</format>
   ```
2. **示例顺序精心设计**（最想模仿的放最后）
3. **核心指令放末尾/独立段落**（避免被前段覆盖）
4. **少用嵌套结构**（Gemini 对深层嵌套理解略弱）
5. **加显式输出格式样板**：
   ```xml
   <example_output>
   # 标题
   - 要点 1
   - 要点 2
   </example_output>
   ```

## 三、按任务类型选模型

| 任务类型 | 首选模型 | 备选 | 提示词设计重点 |
|---------|---------|------|---------------|
| **事实问答** | Claude | GPT | 允许补充相关事实 |
| **创意写作** | Claude | GPT | 少约束，给风格锚定 |
| **代码生成** | Claude | GPT | XML 结构 + 完整 docstring |
| **复杂推理** | Claude Opus | GPT-4.1 | 允许"反驳"和"质疑" |
| **角色扮演** | GPT-4o | Claude | 命令式 + 强约束 |
| **翻译** | Claude | GPT | 直译为主，少"意译发挥" |
| **长文档总结** | Claude | Gemini 1M | 200K 上下文优势 |
| **快速问答** | GPT-4o-mini | Gemini Flash | 速度优先 |
| **数据提取（结构化）** | GPT-4.1 | Claude | JSON schema 显式 |
| **多语言混合** | Gemini | Claude | 多语言原生支持 |

## 四、同一提示词三模型对比示例

### 原始提示词
```
帮我写一篇关于秋天的散文，要写得美一点
```

### 通用优化版（任一模型都可用）
```xml
<role>散文作家，笔触介于川端康成的物哀美学与汪曾祺的市井温情之间</role>
<context>写一篇关于秋天的散文</context>
<constraints>
- 不出现"金风送爽"等陈词
- 以具体场景开篇
- 篇幅 800-1200 字
</constraints>
<format>纯文本，段落间留呼吸感</format>
```

### GPT 定制版
```xml
<role>散文作家，笔触介于川端康成与汪曾祺之间</role>
<context>写一篇关于秋天的散文</context>
<constraints>
- 不出现"金风送爽"等陈词
- 不使用"金风送爽""秋高气爽"等成语
- 不输出"当然可以！""这是一篇关于..."等客套开头
- 不主动总结或升华主题
- 以具体场景开篇
- 篇幅 800-1200 字
</constraints>
<examples>
风格示例（务必参考）：
"梧桐叶落下来的样子，不是飘，是决绝地翻了一个身。"
</examples>
<format>纯文本，段落间留呼吸感，不使用小标题</format>
```
> **关键差异**：显式禁止客套话 + Few-Shot 强制模仿

### Claude 定制版
```xml
<role>散文作家，笔触介于川端康成的物哀美学与汪曾祺的市井温情之间</role>

<context>写一篇关于秋天的散文</context>

<constraints>
- 不出现"金风送爽""秋高气爽"等陈词
- 以一个具体场景开篇（不概括，只白描）
- 情感走向：从感官沉浸到时间感知的微痛
- 篇幅：800-1200 字
</constraints>

<thinking>
不必急着动笔。先在脑中浮现一个具体场景——某个秋日午后，某条街道，某个细节。
从那个细节出发，自然延伸出整篇散文的情感脉络。
避免结构化、避免清单感，让文字像意识流一样流淌。
</thinking>

<examples>
风格示例：
"梧桐叶落下来的样子，不是飘，是决绝地翻了一个身。"
——注意：具体、动感、情感暗线
</examples>

<guidelines>
保留创作自由度。如果觉得结构需要调整，按你认为最自然的方式展开。
但篇幅控制在 800-1200 字。
</guidelines>

<format>纯文本，段落间留呼吸感，不使用小标题</format>
```
> **关键差异**：`<thinking>` 引导意识流 + `<guidelines>` 替代否定句 + 允许 Claude 自主判断结构

### Gemini 定制版
```xml
<role>散文作家</role>

<context>写一篇关于秋天的散文，参考川端康成和汪曾祺的笔触</context>

<task>直接开始写散文。</task>
<format>≤1000 字，3-5 段，**不要前言、不要"以下是..."，第一句就是正文**</format>

<example_output>
梧桐叶落下来的样子，不是飘，是决绝地翻了一个身。
[正文继续...]
</example_output>
```
> **关键差异**：示例放最后（权重高）+ 字数硬约束 + 禁止前言

## 五、模型切换 checklist

当你把同一提示词换到不同模型时，按此检查：

### GPT 检查
- [ ] 显式禁止客套话？
- [ ] 给了 Few-Shot？
- [ ] 指令是命令式（不是请求式）？
- [ ] `<constraints>` 明确"不做什么"？

### Claude 检查
- [ ] 用了 XML 标签？
- [ ] 允许"反驳/质疑"？
- [ ] 复杂任务没给 Few-Shot？
- [ ] 用了 `<guidelines>` 而非否定句？

### Gemini 检查
- [ ] 字数硬约束？
- [ ] 核心指令放末尾/显眼位置？
- [ ] 示例顺序符合"最想模仿的放最后"？
- [ ] 给了完整输出样板？

## 六、跨模型优化 5 条铁律

1. **不要写一版通吃**——至少为首选/备选模型各做一版
2. **角色扮演优先 GPT**——它执行最到位
3. **复杂推理优先 Claude**——它默认就有深度
4. **超长上下文优先 Claude 200K 或 Gemini 1M**——避免切分丢失
5. **JSON/结构化输出优先 GPT-4.1**——对 schema 响应最稳
